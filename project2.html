Participate Shopify storefront

Liquid, CSS, Responsive design

Participate tasked me and a design colleague with managing the company’s Shopify storefront, which features branded swag and classroom items available for purchase through a fulfillment service. I managed the company theme code and website design while my colleague managed photography and new product listings.

Designing the storefront

From the beginning, I was able to manage the appearance of our storefront with Shopify’s theme builder, which is just as robust as Wordpress in terms of the wide level of customization available. Our marketing team ran point with communications and fulfillment, while my team combined our design sense to quickly redesign a storefront that was clean and accessible in the wake of the company rebrand. But I discovered there were some things we wanted to do design-wise that couldn’t be done in the theme builder.

Getting liquid

Shopify storefronts are built with a custom templating language called Liquid that is written in Ruby. I knew little about Ruby going in, but when I got under the hood, I quickly recognized Liquid’s use of HTML elements between Ruby’s curly braces of conditional logic. There were also stylesheets written in the same CSS I was used to. I set up a copy of our store theme as a QA testing site and got busy poking around.

Breaking out of the box

Our theme limited us to three product blocks on our home page. These blocks could be linked to individual products or collections of products, but we didn’t have an immediate way to add more than three. We envisioned a home page that featured all available products in a responsive grid - there weren’t that many different products we had for sale, so categorized collections didn’t make sense. We also wanted control over display properties like borders and padding, which the theme set automatically.

An intermediate photo choice worked well, but it didn't mesh with the default menu. The item grid needed a redesign.

I experimented on the QA theme I’d set up and dissected the index.liquid file that structured our home page. Fortunately, the theme classes were descriptive (a good lesson for future me in writing meaningful class names), and I found the structure responsible for setting our three featured products. Inside a conditional bracket that determined whether featured image settings were active, a parent div controlled each of three divs that contained image and link elements mixed with dynamic liquid tags.

Within a liquid tag, I could add as many new product object copies as we needed

It took some trial and error to discern how certain classes behaved. It became clear that an .alpha and .omega class determined the starting and ending blocks in a row respectively, while the parent container’s .clearfix class allowed subsequent divs to start new rows instead of nesting them within previous ones. Additionally, the properties inside liquid tags needed to be updated each time a new featured item was created so that the pages and images linked together correctly. These new key-value pairs needed to be added in a JSON file to render correctly. There were a lot more moving parts than I expected.

Documentation

After several instances of adding new products, my colleague offered to take on this task if I could teach him how the code was structured. I agreed to write documentation describing what I changed and how to change it.

Writing documentation is a fun test of one’s knowledge, because it just can’t be done effectively if you don’t understand what you’re writing about. I had to gather and present my knowledge in a way that a non-developer could understand. I found that helping a fellow designer learn to update code was best done with color coding (no pun intended).

Snapshot of color-coded documentation written for my colleague

Results

We made design upgrades to the storefront with our new item grid and a cleaner user experience. I take no credit for the excellent product photography!
